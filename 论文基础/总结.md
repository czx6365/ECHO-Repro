给定 Issue / Log / Trace / Codebase，框架先进行相关性检索与环境建模，再生成可执行 harness，并通过执行反馈与 Fail-to-Pass validation 迭代修正，最终得到最小化、可验证的 bug reproduction script。

3. 模块一：Bug Specification Extraction
这一层负责把原始 issue、log、trace 变成结构化 bug spec。
输入：
Issue Description
Error Log
Stack Trace
CI Failure Output
Repo Metadata
Dependency Files
输出：
BugSpec = {
  current_behavior,
  expected_behavior,
  failure_signature,
  exception_type,
  error_message,
  stack_trace_frames,
  environment_clues,
  reproduction_steps,
  oracle_hint
}
这里是你相对 AEGIS / Otter 的扩展点。AEGIS 主要从 issue description 中抽取 current result、expected result、existing reproduction method；你可以进一步加入 Log / Trace / Environment Metadata。
例如：
Issue:
  "ValueError occurs when calling fit() with sparse matrix"

Log:
  ValueError: dimension mismatch

Trace:
  sklearn/linear_model/_base.py:342

Env:
  Python 3.10, numpy 1.24, scipy 1.10
你的 BugSpec 就应该变成：
Current Result: fit() raises ValueError: dimension mismatch
Expected Result: fit() should complete or return correct prediction
Failure Signature: ValueError + dimension mismatch
Suspicious Stack Frame: sklearn/linear_model/_base.py:342
Environment Constraints: Python 3.10, numpy/scipy versions
Oracle: generated harness should fail before patch and pass after patch

4. 模块二：Environment-aware Correlation Retrieval
这是你最重要的创新模块，可以融合 AGENTLESS + iCoRe。
不要只做：
Issue → BM25 / embedding → top-k files
而是做三路检索：
A. Source Code Retrieval
B. Test / Harness Retrieval
C. Environment Retrieval
4.1 Source Code Retrieval
参考 AGENTLESS 的 hierarchical localization：
Repo tree
  ↓
Top-N suspicious files
  ↓
Classes / functions / methods
  ↓
Fine-grained lines / code windows
AGENTLESS 不是一次性把整个 repo 塞给 LLM，而是先定位 suspicious files，再定位 classes/functions，最后定位 edit locations，这个思想适合你的 source code retrieval。

---
4.2 Test / Harness Retrieval
这是 iCoRe 对你最有用的地方。
普通 retrieval 只找 bug file，但 reproduction 还需要知道：
项目原来怎么写 test？
测试入口是什么？
fixture 怎么构造？
mock 怎么写？
dataset / config 从哪里加载？
所以你要单独检索：
related test files
fixture files
conftest.py / pytest.ini / unittest base class
existing regression tests
helper functions
test data files
这正好回应 LIBRO / AEGIS 的常见失败点：生成的 test 逻辑可能对，但无法接入项目测试环境。LIBRO 也指出，失败原因包括缺少 helper definitions、依赖 external resources、误判 expected behavior 等。

---
4.3 Environment Retrieval
这是你的核心差异化。
除了 source 和 test，你还检索：
requirements.txt
pyproject.toml
setup.py
package.json
pom.xml
Dockerfile
docker-compose.yml
tox.ini
pytest.ini
CI yaml
.env.example
config files
data path references
目标不是修 bug，而是回答：
这个 bug 要在哪个环境下跑？
需要安装什么依赖？
需要什么启动命令？
需要什么配置文件？
需要 mock 哪些外部服务？
这部分可以命名为：
Environment-aware Harness Context Retrieval

5. 模块三：Concise Reproduction Context Construction
这一层融合 AEGIS 的 concise context，但你要扩展成五类上下文：
Concise Reproduction Context =
  1. Structured BugSpec
  2. Suspicious Source Context
  3. Related Test Context
  4. Environment Context
  5. Relevance Explanation
也就是给 LLM 的不是一堆乱文件，而是：
## Bug Summary
当前错误是什么，期望行为是什么。

## Failure Signature
错误类型、错误信息、关键 trace frame。

## Suspicious Source Code
可能相关的函数 / 类 / 行号。

## Related Tests and Fixtures
项目已有测试风格、fixture、mock、helper。

## Environment Requirements
依赖、运行命令、配置文件、数据文件路径。

## Relevance Explanation
为什么这些代码、测试、环境与 bug 相关。
AEGIS 证明 concise context 可以减少 agent 的上下文负担，让 Reproducer Agent 更专注于 issue-related code 和 explanation。

6. 模块四：Harness / Test Script Synthesis
这一层融合 Otter + AEGIS + AGENTLESS。
你的输出不只是 test_xxx.py，而是：
Minimal Reproduction Harness =
  reproduce.py / test_reproduce.py
  + execution command
  + dependency setup
  + config/mock/data preparation
  + oracle assertion
生成策略可以分三种 candidate：
Candidate A: Direct unit test
Candidate B: Standalone reproduce.py
Candidate C: Existing-test-style harness
生成 prompt 核心要求
You must generate a minimal executable reproduction harness.

The harness should:
1. Use the project’s existing test style.
2. Include necessary imports, fixtures, mocks, and setup.
3. Trigger the failure signature described in the issue/log/trace.
4. Avoid artificial failures such as `raise AssertionError("failed")`.
5. Produce a real oracle: fail on buggy version, pass on fixed version.
6. Include the exact command to run it.
Otter 对你的启发是：生成 test 后不能直接相信 LLM，需要 rule-based analysis / repair 去检查和修复输出格式、导入、测试结构等。(arXiv)

7. 模块五：FSM-based Execution Feedback Optimization
这一层直接借鉴 AEGIS，但你可以做得更强，因为你有 log / trace / environment feedback。
状态机可以这样设计：
S0 Create Harness
S1 Static Check
S2 Execute Harness
S3 Failure Signature Match
S4 Self-Verify
S5 External-Verify
S6 Modify Harness
S7 Retrieve More Context
S8 Restart
S9 Report
状态转移逻辑
Create Harness
  ↓
Static Check
  ├── syntax/import error → Modify Harness
  └── pass → Execute Harness

Execute Harness
  ├── dependency error → Environment Retrieval
  ├── file/config missing → Environment Retrieval
  ├── unrelated error → Modify Harness
  ├── no failure → Modify Harness / Retrieve More Context
  └── target failure → Failure Signature Match

Failure Signature Match
  ├── error message / trace / output matches issue → Self-Verify
  └── mismatch → Modify Harness

Self-Verify
  ├── pass → External-Verify
  └── fail → Modify Harness

External-Verify
  ├── pass → Report
  └── fail → Modify / Restart
AEGIS 的 FSM 里有 command check、syntax check、patch check、referee verification、failure experience 和 restart，这些都可以迁移到你的 harness synthesis 里。
你的增强是增加：
Dependency Feedback
Environment Feedback
Trace Matching Feedback
Oracle Quality Feedback

8. 模块六：Multi-level Validation & Ranking
这是最终判断“是不是成功复现”的核心。
不能只看：
脚本失败了 = 复现成功
因为可能是：
ImportError
SyntaxError
FileNotFoundError
人为 raise AssertionError
环境缺失导致失败
所以你的验证要分四层。
Level 1：Executable Validation
脚本能不能运行？
依赖是否满足？
命令是否正确？
过滤掉：
SyntaxError
ImportError
ModuleNotFoundError
路径错误
配置缺失

---
Level 2：Failure Signature Validation
检查失败是否和 issue/log/trace 对齐：
exception type match
error message match
stack frame match
output behavior match
current result match

---
Level 3：Fail-to-Pass Validation
最关键：
Buggy Version: fail
Fixed Version: pass
AEGIS 使用 Fail-to-Pass 作为 reproduction script 质量指标，即脚本在当前有 bug 的版本失败，在修复后版本通过。

---
Level 4：Patch Discrimination Validation
如果你接 AGENTLESS，可以进一步验证：
好的 patch 应该让 harness pass
坏的 patch / 原版本应该 fail
AGENTLESS 的 patch validation 就是利用 generated reproduction tests 和 regression tests 选择最终 patch。

10. 你的创新点可以写成 3 个
创新点 1：Environment-aware Harness Synthesis
已有工作多生成 test/reproduction script，你扩展到：
test code + execution command + dependency setup + config/mock/data preparation
也就是从：
Issue → test
升级为：
Issue / Log / Trace → executable reproduction harness
这是最适合你项目的主创新。

---
创新点 2：Correlation-aware Context Retrieval for Reproduction
结合 iCoRe，但加入 environment branch：
source-code correlation
test-case correlation
environment correlation
execution-feedback correlation
比普通 RAG 强，因为它不是只找语义相似文件，而是找“能让 bug 跑起来”的上下文。

---
创新点 3：Feedback-driven Validated Reproduction Loop
结合 AEGIS 的 FSM 和 Otter 的 self-reflective repair：
syntax feedback
runtime feedback
dependency feedback
log feedback
trace feedback
oracle feedback
F2P feedback
最后用 Fail-to-Pass 过滤伪复现。
